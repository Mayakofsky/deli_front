## Подготовка к защите

## 1. Титул

— Здравствуйте, я представляю приложение **DELI** — сервис для учёта совместных расходов.
— Разработано на **Kotlin + Jetpack Compose** с бэкендом на **FastAPI и PostgreSQL**.

---

## 2. Введение

### Идея проекта

DELI решает бытовую проблему: компания друзей куда-то сходила, кто-то заплатил за всех, потом начинается «скинь на карту, а я тебе переводом, а сколько там с меня?». Достаточно создать событие, добавить покупки и участников — приложение само разделит суммы и покажет, кто кому сколько должен.

### Задачи

- Упростить делёж расходов в компании: не считать вручную, не держать в голове
- Автоматический расчёт долгов между участниками с учётом того, кто за что платил
- Хранение истории событий и долгов
- Возможность загружать фото чеков — чтобы не потерять
- Добавление друзей и система уведомлений о новых запросах

### Аналоги

- **Splitwise** — самый известный, но на английском, нет привязки к событиям, нет фото чеков, интерфейс громоздкий
- **Toshl** — личный финансовый трекер, не заточен под групповые расходы
- **Дзен-мани / CoinKeeper** — личные финансы, нет функционала группового деления
- **Сплит (от Т-Банка)** — только для клиентов Т-Банка, нет гостевых пользователей

### Аудитория

DELI — для любого, кто хоть раз делил счёт с кем-то. Студенты, семьи, коллеги по работе, соседи по квартире, путешественники. Любая ситуация, где несколько человек скидываются на общее.

## 3. Демонстрация

### 3.1 Анимация входа и регистрация (~30 сек)

— При запуске появляется экран с анимацией: название DELI плавно появляется, увеличивается, затем подзаголовок «Делим расходы просто». Если приложение открывается не впервые — через 2 секунды автоматически переходит на главную.
— В первый раз пользователь видит кнопку «Начать», нажимает — попадает на регистрацию.
— Сегментированное переключение «Вход / Регистрация», поля: имя, фамилия, почта, пароль. Всё на русском, минимум полей.
— Кнопка отправки: при нажатии появляется спиннер, поля проверяются на пустоту, ошибки сервера выводятся красным текстом.
— После успешной регистрации/входа — сразу переход на главный экран.

### 3.2 Главный экран — три вкладки (~30 сек)

— Вверху — название DELI и аватарка профиля (при клике — профиль).
— Под шапкой — три быстрые карточки: «Событие», «Должник», «Друзья». Каждая — короткий путь к созданию.
— Основная часть — переключаемые вкладки:
  - **«Мне должны»** — люди, которые должны пользователю деньги. Цифры зелёные.
  - **«Я должен»** — кому должен пользователь. Цифры красные.
  - **«События»** — список всех созданных событий с датой и статусом.
— В каждой вкладке — pull-to-refresh, спиннер при загрузке, заглушка если список пуст.

### 3.3 События (~40 сек)

— С главной пользователь нажимает карточку «Событие» — попадает на создание.
— Создание: вводит название, дату, добавляет участников (из друзей или гостей по имени).
— **Чистое событие** — пустой экран с кнопкой «Добавить покупку».
— Добавление покупки: название, сумму, кто получает выгоду (можно всех или конкретных), фото чека.
— **Наполненное событие** — список покупок с суммами и фото, кнопка редактирования/удаления. Сверху — участники. Статус «Активно». Кнопка «Закрыть событие».

### 3.4 Должник (~20 сек)

— Через карточку «Должник» — экран создания: имя, сумму, описание, фото.
— После создания — долг появляется в нужной вкладке.
— При клике — экран деталей: имя, фото, сумма, описание. Красная кнопка «Удалить».

### 3.5 Друзья (~25 сек)

— Четыре вкладки: «Поиск», «Входящие», «Друзья», «Отправленные».
— **Поиск** — строка ввода, поиск пользователей, отправка запроса.
— **Входящие** — запросы от других. Принять (галочка) или отклонить (крестик).
— **Друзья** — список принятых друзей.
— **Отправленные** — запросы, которые отправил пользователь. Можно отменить.
— Переключение — свайпом или по табу.

### 3.6 Профиль (~20 сек)

— Переключение темы (тёмная/светлая), настройка уведомлений, кнопка «Выйти».

### 3.7 Уведомления (~20 сек)

— После входа — фоновая проверка каждые 15 минут через WorkManager.
— Три типа: запрос в друзья, дедлайн долга (3 дня), дедлайн события.
— Повторные уведомления не приходят. Можно отключить в профиле.
— На Android 13+ запрашивается разрешение при первом запуске.

## 4. Описание реализации

### 4.1 Архитектура MVVM

Приложение построено по архитектуре **MVVM** — Model-View-ViewModel.

**Схема:**

```
┌─────────────────────────────────────────────────────────────────┐
│  HomeScreen.kt (View)                                           │
│  homeViewModel.loadData(userId)                                 │
│  homeState — спиннер, список, заглушка                          │
└────────────────────┬────────────────────────────────────────────┘
                     │ подписка на StateFlow
                     ▼
┌─────────────────────────────────────────────────────────────────┐
│  HomeViewModel.kt (ViewModel)                                   │
│  data class HomeUiState(events, owedItems, dueItems,            │
│                         isLoading, error)                       │
│  _uiState.value = HomeUiState(events = data)                    │
└────────────────────┬────────────────────────────────────────────┘
                     │ вызов Repository
                     ▼
┌─────────────────────────────────────────────────────────────────┐
│  EventRepository.kt (Model)                                     │
│  api.listEvents(userId) → HTTP / CacheHelper → SQLite           │
└────────────────────┬────────────────────────────────────────────┘
                     │ HTTP GET
                     ▼
┌─────────────────────────────────────────────────────────────────┐
│  FastAPI → PostgreSQL                                            │
└─────────────────────────────────────────────────────────────────┘
```

**Что сказать:** «View只知道 рисовать интерфейс, ViewModel — хранить состояние и вызывать Repository, Model — доставать данные с сервера или из кэша. Каждый слой занимается своим делом.»

### 4.2 Технологический стек

**Фронтенд:**
- **Kotlin** — основной язык, coroutines, null-safety
- **Jetpack Compose** — декларативный UI, никакой XML-разметки
- **MVVM + StateFlow** — реактивное состояние экранов
- **Navigation Compose** — навигация между экранами
- **Retrofit + Gson** — HTTP-запросы, JSON парсинг
- **Coil** — загрузка изображений (фото профиля и чеков)
- **DataStore** — сохранение настроек темы и уведомлений
- **WorkManager** — фоновые уведомления о долгах и событиях
- **SQLite (CacheHelper)** — локальный кэш данных (офлайн-режим)

**Бэкенд:**
- **FastAPI + Uvicorn** — REST API, асинхронный Python
- **PostgreSQL** — основная база данных
- **SQLAlchemy** — ORM для работы с БД
- **python-multipart** — загрузка фотографий

**Инфраструктура:**
- VDS (Ubuntu), сервер на порту 8000

### 4.3 Как идут запросы

**Пример: загрузка списка событий**

1. Пользователь открыл главный экран
2. HomeScreen вызывает `loadData` у HomeViewModel
3. ViewModel запускает корутину и идёт в EventRepository
4. Repository пытается получить данные через Retrofit
5. Если есть интернет — HTTP GET на FastAPI, SQL-запрос в PostgreSQL, JSON-ответ
6. Если интернета нет — Repository достаёт данные из SQLite-кэша
7. Данные возвращаются: Repository → ViewModel
8. ViewModel кладёт их в StateFlow
9. Compose видит изменение и автоматически перерисовывает экран
